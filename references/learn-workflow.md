# Learn Workflow

对某个子系统做深入调研，把结果写回 `.better-work/`。不是"再跑一次 init"——learn 的粒度是子系统级。

## 什么时候用

- init 结束后，报告里列出"建议后续 learn 的主题"，用户感兴趣
- 即将深度修改某个未记录清楚的模块，先 learn 再动手
- 发现了一个反复踩坑的区域，想建立系统性理解

## 什么时候不用

- 只是想查一两个函数的用法 → 用 Read/Grep 足矣，别走 learn
- 整个项目都不熟 → 先跑 init，别用 learn 当替代
- 子系统已在 `code/danger-zones.md` 或 `shared/map.md` 里讲清楚了 → 没新增价值

## 执行步骤

### Step 1: 锚定主题范围

主题必须可收敛。合格示例：
- `payment` / `auth` / `websocket-gateway` / `config-loader`

不合格示例：
- `backend`（太大）/ `error-handling`（横切关注点，不是子系统）/ `那个有 bug 的地方`（含糊）

### Step 2: 定位主题涉及的文件

```
# 基于关键词的初筛
grep -rl "<topic-keyword>" src/ | head -20

# 如果主题对应某个目录，优先看目录
ls -la src/<topic-dir>/

# 看 import 链（哪些文件依赖这个子系统）
搜索 "from .*<topic>" 或 "import.*<topic>"
```

把相关文件列表限定在 **5-15 个**。超过 15 意味着主题太大，拆分后分多次 learn。

### Step 3: 深度阅读

对这 5-15 个文件：
- 读全文（不是签名扫描）
- 画出内部调用关系
- 找出对外暴露的 API 面（函数/类/endpoint）
- 识别关键数据结构和状态机
- 收集所有 TODO/FIXME/HACK 注释

### Step 4: 提炼发现

写到 `.better-work/` 时按信号类型分流（不要新建 learn-only 的文件）：

| 发现类型 | 写到 |
|---------|------|
| 对外 API 面 + 入口点 | `shared/map.md` 的"任务→文件映射" |
| 子系统内部调用方向 | `shared/map.md` 的"调用关系" |
| 高入度/易错文件 | `code/danger-zones.md` |
| 子系统内反复出现的模式 | `code/conventions.md` |
| 关键 gotchas / "你会这样想但其实不是" | `shared/index.md` 的 Gotchas（如果重要到所有 session 都应知道） |

### Step 5: 提炼一段"子系统概览"

把这次 learn 的 2-3 段核心理解，以一级章节形式追加到 `shared/map.md` 末尾：

```markdown
## <topic> 子系统概览（learn 于 <date>）

[3-5 句描述：做什么、怎么做、与谁交互、关键约束]

### 入口点
- [文件:函数] — [作用]

### 内部关键文件
- [文件] — [作用]

### 与外部的边界
- 上游：[谁调用我]
- 下游：[我调用谁]
- 持久化：[用到什么存储]

### 最重要的 3 件事
1. [...]
2. [...]
3. [...]
```

### Step 6: 报告

向用户展示：
- 读了哪些文件
- 更新了 `.better-work/` 的哪些文件（逐条列出增加的条目）
- 对这个子系统最重要的 3 个判断

## 质量标准

| 项目 | 必须满足 | 不合格 |
|------|---------|--------|
| 主题范围 | 可枚举的 5-15 个文件 | 覆盖 50+ 文件或模糊描述 |
| 每条发现 | 可追溯到具体文件和行号 | "这个子系统很复杂" |
| 写入 | 追加到已有文件，不新建 learn-specific 文件 | 建了 `learn-payment.md` 这种孤立文件 |
| 子系统概览 | 回答"外人看这个子系统应该先知道什么" | 罗列函数清单 |

## 与 update 的区别

| | learn | update |
|---|-------|--------|
| 触发 | 用户主动（想深入某个主题） | 用户主动或会话结束自动 |
| 扫描对象 | 一个子系统 5-15 个文件的全文 | 本次会话的信号 |
| 输出 | 系统性的新知识 | 来自实际踩坑的增量修正 |
| 是否读全文 | 是 | 否 |

Learn 是"带着问题深读"；update 是"基于本次会话痛点查漏补缺"。两者互补。
