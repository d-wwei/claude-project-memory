# Update Workflow

基于本次会话中的工作信号，增量更新项目记忆。

## 核心原则

Update 不是"回顾并总结"，而是**检测信号并响应**。只有明确的信号才触发更新，没有信号就不动。

## 信号检测

回顾本次会话，逐项检测以下信号：

### 信号 1：搜索挫折 → 补充 map.md

**检测**：本次会话中是否发生了"搜索了 3+ 次才找到目标文件"的情况？

**响应**：在 `map.md` 的"任务→文件映射"表中，添加本次的映射关系。

```markdown
# 添加到 map.md

| 要做什么 | 先看这里 | 然后看这里 |
|----------|---------|-----------|
| [本次任务描述] | [最终找到的文件] | [相关联的文件] |
```

### 信号 2：意外测试失败 → 补充 danger-zones.md

**检测**：修改了某个文件后，是否导致了**不在预期内的**其他测试失败？

**响应**：在 `danger-zones.md` 中记录这个依赖关系。

```markdown
# 添加到 danger-zones.md

| 文件 | 入度 | 影响了 | 检查命令 |
|------|------|--------|----------|
| [被修改的文件] | [有多少文件依赖它] | [意外失败的测试所在模块] | [能提前发现影响的搜索命令] |
```

### 信号 3：发现约定 → 补充 conventions.md

**检测**：本次会话中是否发现了"项目中一致遵循的模式"，且该模式之前未记录？

来源：
- code review 中被要求修改的代码风格
- 发现多个文件用同一种方式处理某个问题
- 阅读 lint 规则或 CI 检查发现的约束

**响应**：在 `conventions.md` 中用正确/错误对比对格式记录。

```markdown
# 添加到 conventions.md

## [约定名称]

✓ 正确:
[实际代码示例，标注文件路径]

✗ 错误:
[反面示例]

为什么: [一句话解释]
```

### 信号 4：知识修正 → 更新已有内容

**检测**：本次会话中是否发现了 `.project-memory/` 中已有信息**不再准确**的情况？

来源：
- 重构后文件路径变了
- 约定被新的实践取代
- 危险区域被重新设计，不再危险

**响应**：直接修正对应文件中的过时信息。不保留旧版本（git 会追踪历史）。

### 信号 5：新模块/新功能 → 扩展 map.md

**检测**：本次任务是否新增了模块、服务、或重要的新文件？

**响应**：在 `map.md` 中更新目录结构和模块职责描述。

## 优先级权重

| 信号来源 | 权重 | 理由 |
|----------|------|------|
| 实际踩坑（搜索挫折、意外失败） | 最高 | 来自真实痛苦，确定有用 |
| code review 反馈 | 高 | 来自人类判断 |
| 采样观察到的模式 | 中 | 可能只是局部现象 |
| 推测性结论 | 不记录 | 除非标注 [未验证] |

## 更新流程

1. 扫描本次会话的信号（按上述 5 类）
2. 读取 `.project-memory/` 中对应文件的当前内容
3. 按信号类型执行增量修改
4. 如果 `index.md` 需要更新（新增了重要模块或关键约定），同步更新
5. 检查 `index.md` 是否仍然 ≤ 150 行
6. 向用户报告更新了什么（逐条列出信号和响应）

## 不更新的情况

以下情况**不触发 update**，即使用户执行了命令：

- 本次会话只做了信息查询，没有修改代码
- 所有发现的知识已经在 `.project-memory/` 中记录过
- 唯一的"新知识"是推测性的且无法标注具体来源

此时报告："未检测到新信号，项目记忆保持不变。"
