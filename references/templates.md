# Templates & Quality Standards

每个 `.project-memory/` 文件的模板和质量标准。

---

## cognitive-protocol.md（≤ 30 行）

该项目的认知约束。每次对话通过 `@` 引用自动加载。**纯行为规则，不含任何项目信息。**

可根据项目风险等级选择模板：

### 严格版（支付系统、核心基础设施等高风险项目）

```markdown
# Cognitive Protocol

## 核心原则
- 实事求是：不夸大、不模糊、不隐瞒。没验证过的明确标注"未验证"
- 没有调研就没有发言权：结论基于实际调查，不基于推测

## 完成标准
- 声称完成前：重读原始需求，逐项核对，每项附可观察证据（输出/测试结果/状态变化）
- "改了代码"不是完成，"目标达成 + 证据"才是完成
- "应该可以"是假设不是结论，必须实际验证

## 行为触发器
- 即将说"完成/搞定/done" → 停下，逐项核对需求，列出证据。未满足的明确列出
- 即将说"做不到" → 先列出已尝试的方法，不足 3 种本质不同的策略则继续
- 修改了被多处引用的文件 → 先搜索所有引用方确认影响范围
- 任务有多个部分 → 核对是否每个部分都已交付
- 写了"应该是/大概/一般来说" → 替换为确切信息或标注"需要验证"

## 失败归因
- 归因于环境/工具前，先用具体证据验证（日志、版本、权限）
```

### 标准版（日常业务项目）

```markdown
# Cognitive Protocol

## 完成标准
- 声称完成前逐项核对原始需求，每项附证据
- "应该可以"是假设，需实际验证

## 行为触发器
- 说"完成" → 逐项核对 + 证据
- 说"做不到" → 列出已试过的 3 种不同方法
- 改高引用文件 → 先搜影响范围
- 多部分任务 → 核对完整性
```

### 宽松版（内部工具、实验项目）

```markdown
# Cognitive Protocol

- 声称完成前核对需求完整性
- 改核心文件前搜索影响范围
```

### 质量标准

| 项目 | 必须满足 |
|------|---------|
| 总行数 | ≤ 30 行 |
| 内容 | 纯认知规则，零项目信息 |
| 每条规则 | 可机械检查（agent 能对照自检） |
| 风险匹配 | 项目风险等级越高，约束越严格 |

## index.md（≤ 150 行）

纯项目知识入口。每次对话自动加载。**不含任何认知规则。**

```markdown
# [Project Name]

[一句话：这是什么 + 用什么技术栈 + 怎么部署]

## Module Map (简版)
[顶层目录，每个一句话职责]
调用方向: A → B → C（不可反向）

## Must-Know Rules
1. [违反会导致 bug 或 review 打回的规则]
2. [...]
3. [最多 5 条]

## Testing
[必跑命令]
[条件命令]

## Danger Zones (top 5)
[文件] — [为什么危险，一句话]

## Gotchas
[坑1：你会这样做，但其实应该那样做]
[坑2：...]

## Quick Locate
| 要做什么 | 看这里 |
|---------|--------|
| [任务类型] | [文件路径] |

## Deep Docs
- 完整模块地图: @.project-memory/map.md
- 编码约定: @.project-memory/conventions.md
- 危险区域详解: @.project-memory/danger-zones.md
- 任务进度: @.project-memory/progress.md
```

### 质量标准

| 项目 | 必须满足 | 不合格示例 |
|------|---------|-----------|
| Module Map | 每个顶层目录都有职责说明 | 只列了目录名没有说明 |
| Must-Know Rules | 每条是"违反了会怎样"而非"建议怎样" | "建议用 structlog" |
| Danger Zones | 每条有文件路径 + 为什么危险 | "数据库相关的要小心" |
| Quick Locate | 每条有具体文件路径 | "在 services 目录下找" |

```markdown
# Module Map

禁止：[列出不允许的调用方向]

## 高频修改文件 (git 热点)

| 要做什么 | 第一步看 | 相关文件 | 测试在 |
|----------|---------|---------|--------|
| ... | ... | ... | ... |

| 模块 | 负责人/团队 | 沟通方式 |

## conventions.md

编码约定，用"正确/错误对比对"表达。

```markdown
# Coding Conventions

## [约定类别 1：错误处理]

✓ 正确:
```[语言]
# [文件路径:行号] — 示例
[实际正确代码]
```

✗ 错误:
```[语言]
[常见错误做法]
```

为什么: [一句话解释原因]

---

## [约定类别 2：数据库访问]
[同样格式]
```

### 质量标准

| 项目 | 必须满足 |
|------|---------|
| 每条约定 | 有正确示例 + 错误示例 + 为什么 |
| 正确示例 | 引用实际项目代码（文件路径:行号） |
| 错误示例 | 是 agent 真的可能会犯的错，不是极端情况 |
| 为什么 | 解释后果（"会导致..."），不是偏好（"我们更喜欢..."） |

## danger-zones.md

高风险区域详解。每个条目必须是可操作的。

```markdown
# Danger Zones

## [文件路径]

- **入度**: [N] 个文件依赖此模块
- **依赖方**: [列出主要依赖方目录/文件]
- **为什么危险**: [具体解释]
- **改之前做**: [具体的检查步骤/命令]
- **历史事故**: [如果有，描述之前因为改这里导致的问题]

---

## [下一个文件]
[同样格式]
```

### 质量标准

| 项目 | 必须满足 | 不合格 |
|------|---------|--------|
| 入度 | 来自实际 import 分析 | 凭感觉说"很多地方用" |
| 检查命令 | 可直接执行的搜索命令 | "检查一下影响范围" |
| 为什么危险 | 说明改错的具体后果 | "这个很重要" |
